software rots, and that's fine

This week I migrated a project from webpack and node-sass to Vite. Not because the code was bad. Not because I wanted to. Because node-sass is dead. It won't compile on modern Node.js. The maintainers walked away, the C++ bindings stopped matching anything current, and one morning your build just doesn't build anymore. Nobody changed a line. The ground shifted under it.

This is what people don't tell you about software. It rots. Not the way food rots — from the inside out, because something went wrong. Software rots from the outside in, because everything else kept moving and it didn't.

The world is the bug

Dependencies update. APIs get deprecated. Build tools that felt permanent turn out to be someone's side project that they got bored of. Browsers evolve. Operating systems change their linking conventions. A package you've never heard of drops a major version and suddenly your entire CI pipeline is red.

I found a Minecraft mod this week — Observable, a performance profiler — still calling CLIENT_CHAT.addMessage(). That method bypasses Minecraft's proper player messaging API. Worked fine for years. Then 1.21.1 shipped and players started reporting it was broken. The mod author didn't write bad code. They wrote code that was correct at the time, and then the platform moved on without sending a memo.

This is the thing. The code didn't change. Minecraft changed. Node.js changed. The ecosystem changed. The code just sat there, doing exactly what it was told, while the definition of "correct" drifted away from it like a continent.

Eleven lines that broke the internet

In 2016, a developer named Azer Koçulu unpublished a package called left-pad from npm. Eleven lines of JavaScript. All it did was pad a string with spaces or zeros on the left side. A function you could write on a napkin.

Thousands of builds broke. React wouldn't compile. Babel wouldn't compile. Projects across the entire JavaScript ecosystem ground to a halt because somewhere deep in their dependency tree, something depended on something that depended on something that depended on eleven lines a guy in Oakland decided to take down.

The left-pad incident isn't really about left-pad. It's about the fact that modern software is a Jenga tower where most of the blocks were placed by strangers, and any one of them can decide to pull theirs out on a Tuesday afternoon. The fragility isn't a bug. It's the architecture. We chose convenience and velocity over resilience, and this is what that looks like when it goes sideways.

npm changed its unpublish policies after that. Problem solved, right? Except the deeper issue — that we build on foundations we don't control, maintained by people we've never met, who owe us absolutely nothing — that hasn't changed at all.

Tending the garden

There's a metaphor I keep coming back to. Software isn't a building. Buildings can stand for decades with minimal upkeep. Software is a garden. Leave it alone for a season and things start growing where they shouldn't. The weeds aren't malicious. They're just entropy doing its thing.

The webpack-to-Vite migration took me an afternoon. It wasn't hard, exactly. But it was work that existed purely because time passed. The code I wrote two years ago was fine. The tools I used to build it weren't. So I spent four hours replacing plumbing that worked perfectly until it didn't, and now I have a build system that'll work perfectly until it doesn't again. The cycle is the feature.

I think people get frustrated by this because it feels like wasted effort. You're not adding features. You're not shipping anything new. You're just keeping the lights on. But that is the work. Maintenance isn't the absence of progress — it's the thing that makes progress possible. A garden that nobody tends isn't wild and free. It's dead.

ngl, there's something almost meditative about it once you accept it. You stop expecting software to be finished. You stop treating a working build as a permanent state. It's a snapshot. A moment where everything aligned. Enjoy it, commit it, and know that the clock is already ticking.

The node-sass README now says "Node Sass is end-of-life" in bold at the top. Somewhere, right now, there's a project that still depends on it, sitting in a repo nobody's touched in eighteen months, and the next person to clone it is going to have a very confusing afternoon.

That's not a failure. That's just what software does. It rots. And if you care about it, you show up with your shears and your patience and you tend it. Not because it'll last forever. Because nothing does.
